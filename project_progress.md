### 2026-06-12 — Phase B hardening: close `forceDelete()` bypass on system pages

- **What changed**: A post-review audit found that `App\Models\Page`'s `is_system` protection guards (`deleting`/`updating`) did not cover Laravel's separate `forceDeleting` lifecycle event, so `->forceDelete()` could hard-delete an `is_system` page without triggering any guard. Added a `static::forceDeleting()` listener in `Page::boot()` that throws `\RuntimeException('System pages cannot be force-deleted.')` for `is_system` pages, mirroring the existing `deleting` guard. Also enabled the in-memory SQLite testing database (`phpunit.xml`, previously commented out) so the test suite no longer points at the real MySQL dev database (`poised_cms`), and added `tests/Unit/PageSystemProtectionTest.php` covering all `Page` model protection rules: system-page delete/force-delete/slug-change/template-change all throw; non-system page slug/template changes after creation throw; non-system page soft-delete, force-delete, and other-field updates succeed normally.
- **Files modified**: `app/Models/Page.php`, `phpunit.xml`.
- **Files created**: `tests/Unit/PageSystemProtectionTest.php`.
- **Database changes**: none.
- **Routes**: none added/changed.
- **Packages**: none added.
- **Why**: closes a protection gap identified during the Phase B architecture review before formal close-out, per explicit request not to defer it to Phase F.
- **Verified**: `php artisan test` — 11 passed (18 assertions), including all 9 new `PageSystemProtectionTest` cases and the 2 pre-existing example tests.
- **Rollback**: revert `app/Models/Page.php` (remove the `forceDeleting` listener), revert `phpunit.xml` (re-comment the `DB_CONNECTION`/`DB_DATABASE` testing env lines), delete `tests/Unit/PageSystemProtectionTest.php`.

### 2026-06-21 — Menu items can now link directly to a Product, Blog Post, or News Article

- **What changed**: user asked how Products/Blog/News would be added to site menus once created. Investigation found two real gaps in the existing Menu/MenuItem admin UI: (1) the "URL" field used Laravel's strict `'url'` validation rule, rejecting site-relative paths like `/products` (same bug class as the `url` *template field* type fixed 2026-06-18, just never applied to menu items specifically — they're validated by a separate, unrelated controller); (2) there was no way to pick a Product/Blog Post/News Article by name the way there already was for Pages — an admin would have had to know/type the exact full URL by hand.
  - **Validation fix**: extracted the existing `HandlesTemplateFields::urlOrPathRule()` closure (accepts absolute URL, relative path, or mailto:/tel:) into a new standalone `App\Http\Controllers\Admin\Concerns\ValidatesUrlOrPath` trait — `HandlesTemplateFields` now `use`s it instead of duplicating the logic, and `MenuItemController` adopted it too, replacing the strict `'url'` rule on both `store()` and `update()`.
  - **Content pickers, no schema changes**: rather than adding new polymorphic columns to `menu_items`, the existing `url` (plain string) column does double duty — `Product`/`BlogPost`/`NewsArticle` each gained a new `url()` method (mirroring `Page::url()`), and the menu item form gained a "Link To" type selector (Custom URL / Page / Product / Blog Post / News Article). Picking a Product/Post/Article from its dropdown copies that resource's `url()` into the same `url` input via a small inline script (`admin/menus/edit.blade.php`'s `@push('scripts')`), extracted per-field-group into a new `admin/menus/partials/link-fields.blade.php` partial shared by both the "add new item" and every "edit existing item" form on the page. Re-opening an existing item's form re-detects which tab to show (Page if `page_id` is set; Product/Blog Post/News Article if `url` exactly matches one of those resources' resolved URLs; Custom URL otherwise).
  - **Why `Product::url()`/`BlogPost::url()`/`NewsArticle::url()` return *relative* paths (deliberately different from `Page::url()`, which returns absolute)**: a Page-linked menu item re-resolves `$item->page->url()` fresh on every page render, so an absolute URL is always current. A Product/Blog/News-linked menu item, by contrast, has no FK — the resolved URL is baked into the plain `url` column once, at save time. Returning an absolute URL there would silently break every such menu link on a future domain change; a relative path doesn't have that failure mode and works identically in an `<a href="">`.
  - `MenuController::edit()` now also loads `$products`/`$blogPosts`/`$newsArticles` (all of them, any status — same as the existing unfiltered Page dropdown) and passes them to the view.

### 2026-06-21 — Fixed validation errors (and a related value-blanking bug) leaking across menu item forms

- **What changed**: user reported that submitting the empty "Add Menu Item" form on `/admin/menus/{id}` showed validation errors not just on that form, but appeared to attribute them to already-saved menu items too. Root cause: every item's update form and the "Add Item" form live on the same page and use identical field names (`label`/`url`/`page_id`/`target`), but Laravel's default validation error bag and `old()` input are both **global to the session**, not scoped per-form — so a failure on any one form lit up `@error()` directives with the same field name on every other form on the page. This got measurably worse the same day the Product/Blog/News link pickers were added (2026-06-21, earlier session): `link-fields.blade.php`'s new inline `@error('url')`/`@error('page_id')` calls are `@include`d by *every* existing item's form too, so a failed Add-form submission started visibly displaying "must be a valid URL"/"required" text inside every other item's own Link fields — not just a single ambiguous summary line as before.
  - **A second, related bug found while fixing the first**: `old('label', $item->label)` doesn't care which form actually submitted — if the failed Add form posted `label=''`, *every* existing item's Label input would silently render blank too (the key `'label'` exists in the flashed old-input array, even though its value is empty, so `old()` returns `''` instead of falling through to `$item->label`). This wasn't part of the user's report (no error text appears for a blanked field, so it's easy to miss) but is the same root cause and would have been just as confusing — an item's label looking wiped out for no visible reason.
  - **The fix — named error bags**: `MenuItemController::store()`/`update()` now use `$request->validateWithBag(...)` instead of plain `validate()`. The "Add Item" form gets a fixed bag (`MenuItemController::NEW_ITEM_ERROR_BAG`); each existing item's form gets a bag unique to that item's ID (`MenuItemController::errorBagFor($menuItem)`, e.g. `'menuItem42'`). Every `@error('field')` in `edit.blade.php` and `link-fields.blade.php` now passes the matching bag name (`@error('field', $bagName)`), and the page's old generic `@if ($errors->any())` summary was replaced with one summary box per form, scoped to that form's own bag — matching the summary-plus-inline convention already used by every other admin edit/create form in this codebase (confirmed by checking 15 other admin views).
  - **The fix — old() value scoping**: since `validateWithBag()` doesn't change how `old()`/`withInput()` work (confirmed by reading Laravel's source — it's a thin wrapper that only tags which bag the *error messages* land in), each form now computes its own `$oldFor($bag)` — a closure that only consults `old()` when that specific form's own bag actually has errors, otherwise it just uses the real saved value. Applied everywhere a field could be re-populated: label, target, the URL/Page/Product/Blog Post/News Article picker fields, and the is_active checkbox.
