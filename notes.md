# Developer Notes

_Compiled from a full codebase audit on 2026-06-12. These notes capture observations and decisions inferred from the actual implementation — not aspirational design._

## Page CRUD / dynamic pages / revisions (Phase L, 2026-06-19)

- **Two page-rendering pipelines now coexist on purpose.** The 5 system pages (`home`/`about`/`services`/`solutions`/`contact`) render via hand-written Blade files in `resources/views/pages/*.blade.php`, each with its own named route and `PageController::show($slug, $view)` call in `routes/web.php`. Any other page (created via the admin) renders via the **generic** pipeline: `PageController::showDynamic($slug)` → `resources/views/pages/dynamic.blade.php` → one `@includeIf` per section into `resources/views/partials/dynamic-sections/{section_key}.blade.php`, reached only via the catch-all `Route::get('/{slug}', ...)` that must stay the **last** route in `routes/web.php`. Don't try to unify these — the system pages' hand-written views exist precisely because their layouts are bespoke/non-generic (carousels, video backgrounds, image collages), and forcing them through the generic pipeline would mean rebuilding them as section types, which is exactly the kind of unnecessary refactor the user explicitly ruled out.
- **A new section type used by a future template needs one new file, not a framework change.** Add the section to `config/cms/templates.php` as usual, then add `resources/views/partials/dynamic-sections/{section_key}.blade.php` following the existing partials' pattern (`Content::field()`/`items()`/`itemField()`, reuse existing Bootstrap classes for visual consistency). `dynamic.blade.php`'s `@includeIf` means a missing partial is silently skipped, not a hard error — useful while building one out, but means a typo'd section_key will just render nothing with no warning. Check this first if a new section "doesn't show up."
- **`PageRevisionService::record()` must snapshot a freshly-fetched model, never the in-memory one right after `Page::create()`.** DB column defaults (e.g. `pages.robots` defaults to `'index,follow'`) aren't hydrated back into an Eloquent model after `::create()` — the in-memory instance has `null` for any column you didn't explicitly set, even though the DB row has the real default. Snapshotting that captures `null`, and restoring it later writes `null` into a NOT NULL column → `QueryException`. This was a real bug, only caught by manually walking the full create→edit→restore cycle against the live dev DB — the automated tests didn't catch it because they always passed explicit values for every column. **Lesson: when adding any future snapshot/audit/history feature, always re-fetch from DB before serializing, never trust the in-memory model state right after a partial `create()`.**
- **Publish state (`status`/`published_at`) is deliberately excluded from what `restore()` writes back**, even though it's still captured in the snapshot for visibility. Found via the same manual walkthrough: restoring an old revision (taken while a page was still `draft`) silently unpublished a page that had since gone live — a "restore my content" action quietly took a live page offline. Treat publish/workflow state the same as structural state (which sections/items exist) — both are explicitly out of scope for what a content-revision rollback should touch. If a future "full rollback including publish state" feature is ever requested, it needs its own explicit, clearly-labeled action — never bundle it into the default restore.
- **`route($somePage->slug)` is not a safe way to link to an arbitrary page** — it only worked for the 5 system pages because their route names happen to equal their slugs (and even then, services/solutions use singular URL *paths* — `/service`, `/solution` — while their slugs and route names stay plural). Use `$page->url()` (added to the `Page` model) for any future code that needs to link to a `Page` by model, and `$page->hasNamedRoute()` if you need to branch behavior (e.g. nav "active" state) between system and dynamic pages.
- **Manual verification against the real dev server caught two real bugs the test suite didn't.** Both bugs above passed all 188 automated tests (run against the in-memory SQLite test DB) before being found by hand-walking create→publish→edit→restore→delete through actual HTTP requests against the live MySQL dev DB. The gap: tests always provided complete, explicit field values, so they never exercised the "DB default not yet hydrated" path, and no test happened to restore-after-publish in that exact order. When building anything that touches model defaults or has an unusual operation ordering (snapshot/restore, migrations-that-backfill, etc.), a manual walkthrough of the real sequence a user would actually perform is worth doing even with full test coverage.

## Architecture decisions

- The project is a **plain Laravel MVC app with no extra layers** — no services, repositories, policies, or domain modules. If you're tempted to add a service layer "to be consistent," there's nothing to be consistent with yet; keep new code simple and Laravel-idiomatic until a real need for abstraction appears.
- Public marketing pages are implemented as **route closures returning views directly** (no controllers). If these pages start needing logic (e.g., dynamic content from a `pages` table), introduce a controller at that point rather than cramming logic into `routes/web.php`.
- The admin area uses a **separate layout and asset set** (`admin/layouts/app.blade.php`, `public/admin/assets/`) from the public site (`layouts/app.blade.php`, `public/assets/`). Don't mix these — admin views should not pull in public site partials and vice versa.

## Database observations

- As of Phase B, the CMS schema exists: `media`, `pages`, `page_sections`, `section_fields`, `section_items`, `item_fields`, `menus`, `menu_items`, `settings`, `cache_versions` — see "Phase B notes" below for the field-based architecture.
- Default connection is **SQLite** (`database/database.sqlite`, local dev file). `config/database.php` also defines a MySQL connection for non-default use.
- Session, cache, and queue all use the `database` driver (per `.env.example`), meaning the `sessions`, `cache`/`cache_locks`, and `jobs`/`job_batches`/`failed_jobs` tables are actively used by the framework even though no app-specific tables exist yet.
- `DatabaseSeeder` seeds exactly one user: `test@example.com` (via `UserFactory`, password is the factory default "password", hashed). The `User::factory(10)->create()` line is commented out.

## Phase B notes (2026-06-12) — CMS schema architecture

- **Field-based, EAV-style content storage**: `page_sections` and `section_items` are *purely structural* (which `section_key`/`item_type`, which page/section, order, active flag). All actual content is rows in `section_fields` (`page_section_id`, `field_key`, `value`, `media_id`) and `item_fields` (`section_item_id`, `field_key`, `value`, `media_id`). This means **new section/item types never require a migration** — only a new entry in `config/cms/templates.php` plus a Blade partial. `PageSection::field($key)` / `SectionItem::field($key)` helper accessors return `value` or the related `Media` model depending on whether `media_id` is set.
- **Template Registry** lives at `config/cms/templates.php` and is loaded via `AppServiceProvider::register()` using `mergeConfigFrom(config_path('cms/templates.php'), 'cms.templates')` — Laravel does **not** auto-load config files in subdirectories of `config/`, so this merge is required. Access via `App\Cms\TemplateRegistry` (`pageTemplates()`, `allowedSections()`, `sectionFields()`, `itemFields()`, etc.). Any `section_key`/`item_type`/`field_key` written to the DB should be validated against this registry at the application layer (DB has no type enforcement on `value`/`media_id`).
- **`pages.is_system`**: the 5 seeded pages (`home`, `about`, `services`, `solutions`, `contact`) have `is_system=true`. `App\Models\Page` boot-time guards throw `\RuntimeException` if code tries to (a) delete an `is_system` page, (b) change `slug`/`template` on an `is_system` page, or (c) change `slug`/`template` on *any* existing page after creation (these two fields are creation-time-only for custom pages). Any future "New Page"/"Edit Page" admin controller must catch/avoid triggering these, not bypass them.
- **Media**: `media` table is schema-only (Phase C will add upload/storage/picker). All image/logo/favicon references across `pages`, `section_fields`, `item_fields`, and `settings` use a `media_id` FK (`ON DELETE SET NULL`), never path strings.
- **Soft deletes**: `media`, `pages`, `page_sections`, `section_items` have `deleted_at`. Note Eloquent soft-delete does **not** cascade to children automatically (only hard-delete FK cascade does) — a future "delete page" admin action must explicitly soft-delete its sections/items too if a cascading trash UX is wanted.
- **No Blade views were changed in Phase B** — `page_sections` rows exist for all 5 system pages (counts: home=8, about=3, services=4, solutions=5, contact=3) per their template's `allowed_sections`, but `section_fields`/`section_items`/`item_fields` are empty. Phase I will populate these per-section (transcribing from the existing Blade partials) and rewire each partial to read from the DB, one section at a time, checking visual parity.
- **Menus**: `menus`/`menu_items` tables + seed data exist (`header`/`footer`, each with 5 items linking to the 5 system pages via `page_id`). No admin CRUD yet — deferred to a later phase as agreed.
- **Settings**: `settings` table seeded with 11 `theme`-group rows (9 color hex values + `logo`/`favicon` as `media_id` placeholders, currently `null`). No admin UI yet.

## Phase C notes (2026-06-12) — Media Library

- **Storage**: uploads go to the `public` disk, under `storage/app/public/media/`, served via the `public/storage` symlink (created by `php artisan storage:link` as part of this phase — did not exist before). `App\Http\Controllers\Admin\MediaController::store()` validates `file` (`required|file|mimes:jpg,jpeg,png,gif,svg,webp,pdf|max:5120`) and creates a `Media` row with `disk='public'`, `path` = the stored path, plus `filename`/`mime_type`/`size` from the uploaded file.
- **Soft delete vs force delete (lifecycle hardening)**: `Media::destroy()`-equivalent (the admin "Delete" button → `MediaController::destroy()`) calls `$media->delete()` — **database-only soft delete, file stays on disk**, so the item remains restorable via `Media::restore()`. A new `Media::boot()` `forceDeleting` listener (mirroring `Page`'s `deleting`/`forceDeleting` pattern) deletes the physical file from `Storage::disk($media->disk)` only when a `Media` row is **force-deleted** — this hook exists now so any future "permanently delete" admin action (not yet built) gets correct file cleanup for free.
- **No admin restore/trash UI yet**: Phase C only ships the soft-delete ("move to trash") action. Restoring or permanently deleting media has no UI — deferred to whenever a "Trash" view is needed. The `forceDeleting` cleanup hook means that UI addition will be purely additive (view/route/controller-action), no further model changes required.
- **Media reuse across pages/settings — open consideration for Phase F/I**: `media_id` FK columns (`pages.og_image_id`, `section_fields.media_id`, etc.) are all still `null` (per Phase B). When Phase F/I wires up `belongsTo(Media::class)` relations and media pickers, decide whether those relations need `withTrashed()` (so a soft-deleted-but-still-referenced media item keeps rendering) and/or whether the admin "delete" action should warn if a media item is still referenced. Not an issue today since nothing references `media_id` yet.
- **No orphaned-file cleanup job**: not needed for Phase C (upload creates the file and the `Media` row in the same request, no intermediate failure window). If disk/DB drift becomes a real problem later, a scheduled `artisan media:prune-orphans`-style command would be the fix — noted here for future reference only.
- **New sidebar link**: "Media Library" added to `admin/partials/sidebar.blade.php` (between Dashboard and the still-dead "Home CMS" placeholder), using the same `routeIs(...) ? 'active' : ''` pattern as Dashboard. This is the first previously-dead sidebar slot to become a real link.

## Phase D notes (2026-06-12) — Theme Settings

- **Edit-only, fixed-schema**: `SettingController` only ever updates the 11 existing `theme`-group `settings` rows (9 colors + `logo`/`favicon`) seeded by `SettingsSeeder`. It never creates new `settings` rows. If a future need arises for arbitrary/dynamic settings, that requires its own registry-based design — don't extend this controller for it.

Per the Phase D architecture review, a reusable media picker should be built in Phase F where `section_fields.media_id`/`item_fields.media_id` create multiple real consumers — building it now (for 2 fields) would mean designing it against a single low-stakes use case and likely rebuilding it later.
- **Soft-deleted media referenced by settings**: `Setting::media()` uses the default (non-trashed) scope — if a media item assigned as logo/favicon is soft-deleted, the relation returns `null` and the settings view simply shows no preview/falls back to none. No `withTrashed()`, no cascading "clear reference on delete" listener. This is an acceptable, recoverable state (admin can restore the media or pick a new one via `Media::restore()`).
- **Open follow-up (not blocking, not yet built)**: now that `logo_media_id`/`favicon_media_id` are real references, `MediaController::destroy()` could optionally warn an admin if the media item being deleted is currently assigned as the site logo/favicon (a simple `Setting::where('media_id', $media->id)->exists()` check). Deferred — no UI currently surfaces this risk, and soft-delete keeps it recoverable either way.
- **Nothing reads these settings yet**: as with Phase B, `settings` values are still not consumed anywhere on the public site — Phase I will wire `partials/head`/`partials/topbar`/etc. to read theme colors and logo/favicon from these rows.
- **New sidebar link**: "Theme Settings" added to `admin/partials/sidebar.blade.php` between "Media Library" and the still-dead "Home CMS" placeholder, using the same `routeIs(...) ? 'active' : ''` pattern.

## Phase F notes (2026-06-14) — Page & Section CRUD

- **`section_fields`/`item_fields` rows are created on first save, not pre-seeded**: `PagesSeeder` only creates structural `page_sections`/`section_items` rows (Phase B) with no field content. `PageSectionController::update` and `SectionItemController::store`/`update` use `updateOrCreate(['field_key' => $key], [...])` via `HandlesTemplateFields::syncTemplateFields()`, so the first edit of any section/item creates its `section_fields`/`item_fields` rows; subsequent edits update them in place. Any future code reading `section_fields`/`item_fields` (e.g. Phase I) must not assume a row exists for every `TemplateRegistry` field — use `PageSection::field($key)`/`SectionItem::field($key)` (return `null` if absent), not direct relation access.
- **Media `<select>` empty-value handling**: HTML `<select>` "— None —" options submit `""`, not absent. Laravel's `nullable` validation rule does **not** treat `""` as `null` for the `exists` rule, so `HandlesTemplateFields::normalizeMediaFields()` rewrites `fields[$key] === ''` to `null` (for `media`-type fields) and `PageController::update` does the same for `og_image_id` *before* `$request->validate()` runs. Any new media-type field/select must go through this same normalization or `exists:media,id` will incorrectly reject "no selection".
- **Unknown/unregistered `section_key` handling**: `TemplateRegistry::sectionFields()`/`itemSchema()` return `[]`/`null` (never throw) for a `section_key` not present in `config/cms/templates.php`. `admin/pages/edit.blade.php` checks `TemplateRegistry::section($section->section_key)` and shows a "not defined in the Template Registry" notice with no field inputs in that case; `PageSectionController::update` still runs (empty `$fieldDefs` → empty rules → nothing to sync) and updates `is_active` only. This means a stray/legacy `section_key` can never crash the Pages admin — confirmed by `test_unknown_section_key_is_handled_gracefully`.
- **Section item ordering**: `SectionItemController::move` swaps `order_column` between the target item and its immediate previous/next sibling in `$section->items` (already ordered by `order_column` via the `PageSection::items()` relation). New items get `order_column = max(existing) + 1` (or `0` if none exist). There is no drag-and-drop and no renumbering/compaction — gaps or ties in `order_column` are tolerated since ordering is always relative (`orderBy`), not absolute-index-based.
- **`og_image_id` and `media`-type fields reuse a single generic partial**: `resources/views/admin/partials/media-select.blade.php` lists *all* `Media` rows (any mime type), unlike Phase D's settings dropdown which filters to `image/%`. This is intentional — `TemplateRegistry` `media`-type fields (e.g. `hero.background_image`, `services_grid` item `icon` is a string not media, but `testimonial.photo`, `brand-logo.logo`) are documented as images in practice but the schema doesn't enforce a mime restriction, and `og_image_id` could reasonably reference a non-image (e.g. a PDF) in some OG setups. If a future requirement needs image-only enforcement per field, add a `mime` key to the relevant `config/cms/templates.php` field definitions and branch in `media-select.blade.php`/validation accordingly — don't special-case it ad hoc.
- **`richtext` field (`content.body`) still has no sanitization**: Phase F stores it as a plain `<textarea>` → `longText` value, same as `text` fields — no WYSIWYG, no HTML sanitization on input or (eventual) output. This remains an open item for whichever phase first renders `richtext` content on the public site (Phase I) — flagged in the Phase F architecture review as a pre-existing concern, not introduced by this phase.
- **Validation lives entirely in controllers, not Form Requests**: consistent with the project's existing inline-`$request->validate()` convention (`AuthController`, `SettingController`). `HandlesTemplateFields` is a plain trait shared by `PageSectionController`/`SectionItemController` to avoid duplicating the `TemplateRegistry`-driven rule-building and field-sync logic — not a service layer; don't extend this pattern beyond these two controllers without re-evaluating.
- **New sidebar link**: "Pages" replaces the 4 dead "Home/About/Solutions/Services CMS" links in `admin/partials/sidebar.blade.php`, using the same `routeIs('admin.pages.*') ? 'active' : ''` pattern as other sidebar links. "Contact Messages" remains the only dead link (Phase H).
- **Out of scope (explicitly deferred)**: creating new pages, deleting pages, adding/removing sections from a page's `allowed_sections`, drag-and-drop reordering, and any frontend rendering changes — none of these were touched in Phase F.

## Phase B hardening notes (2026-06-12)

- **`Page` model protection is now complete across both Eloquent delete lifecycles**: `static::deleting` blocks `->delete()` (soft delete) and `static::forceDeleting` blocks `->forceDelete()` (hard delete) for `is_system` pages — both throw `\RuntimeException`. `static::updating` still blocks slug/template changes on `is_system` pages, and blocks slug/template changes on *any* page after creation.
- **Remaining bypasses (documented, not yet closed — deferred to Phase F per architecture review)**: query-builder mass operations (`Page::query()->delete()`, `Page::where(...)->update(...)`) and raw DB access (`DB::table('pages')->...`) skip Eloquent model events entirely and are NOT protected by these guards. Any future bulk-admin action must load-and-iterate model instances (`$pages->each->delete()`), never use query-builder mass mutations on `pages`.
- **Recommended for Phase F**: add `Page::deletable()` scope (`where('is_system', false)`) and use it for any admin page-list "select & delete" UI, so `is_system` pages are never offered as deletable. An `editable()` scope was considered but judged low-value since the `updating` guard already enforces slug/template locks per-instance.
- **Testing DB**: `phpunit.xml` now sets `DB_CONNECTION=sqlite` / `DB_DATABASE=:memory:` for the `testing` environment (previously commented out, which meant `php artisan test` would have run `RefreshDatabase` against the real MySQL dev database `poised_cms` per `.env`). All future tests using `RefreshDatabase`/`DatabaseTransactions` now run safely against an in-memory SQLite DB recreated from the full migration set.
- **New test coverage**: `tests/Unit/PageSystemProtectionTest.php` — 9 tests covering every `Page` model protection rule (system-page delete/force-delete/slug/template guards, non-system page creation-time-only slug/template lock, and confirming normal soft-delete/force-delete/field-update still work for non-system pages).

## Phase E notes (2026-06-14) — Schema Alignment

- **`hero` is additive, not redesigned**: the original 5 single-value fields (`heading`, `subheading`, `button_text`, `button_url`, `background_image`) remain exactly as they were — about/services/solutions/contact still use them for a single hero banner with no slides. The new `autoplay`/`interval` fields and the `hero-slide` item schema are *layered on top*. A page's hero is only a carousel if it has `hero` items (home has 3, seeded structurally); pages with zero hero items just render their single-hero fields as before. Phase I's hero partial will need to branch on `$section->items->isNotEmpty()` to pick carousel vs. single-banner rendering.
- **No `boolean` field type exists** (confirmed by reading `HandlesTemplateFields::templateFieldRules()` — only `string`/`text`/`richtext`/`url`/`integer`/`media` are recognized, anything else falls through to `string|max:255`). `autoplay` is therefore `type: integer` (1/0), not a real checkbox — if a future phase adds a `boolean` type to `HandlesTemplateFields` + `field-input.blade.php` (checkbox rendering + `boolean` validation rule), `autoplay` should be migrated to it then. Not done now to keep Phase E config-only/non-invasive to the admin UI.
- **Four new reusable section/item types** were added, each designed to be shared across multiple pages rather than per-page bespoke types:
  - `stats` / `stat` (`label`, `value` — both plain strings so values like "100+", "99%", "24/7", "PAN India" all fit) — used by home, about, services.
  - `faq` / `faq-item` (`question` required, `answer` required text) — used by contact.
  - `process_steps` / `process-step` (`step_number` string e.g. "01", `title`, `description`) — used by solutions.
  - `checklist` / `checklist-item` (`text` required, optional `description`, optional `icon` class string) — used by about, services, solutions. This replaces what would otherwise be ad-hoc bullet lists; `richtext` was deliberately avoided per the task spec ("Do NOT use richtext as a workaround").
- **`landing_page` (solutions) `allowed_sections` was a near-total mismatch before this phase** (`features, brand_logos, testimonials, cta` — none of which correspond to the actual Solutions page, which has a hero, intro content, a checklist of differentiators, a 6-card solutions grid, and a 4-step process). It is now `hero, content, checklist, services_grid, process_steps, cta`. The 6-card "Solutions" grid reuses the existing `services_grid`/`service-card` item schema (same `title`/`description`/`icon`/`link_url`/`link_text` shape) rather than inventing a `solutions_grid` type — deliberate minimal-change reuse.
- **`contact_info` dual phone/email**: added `phone_secondary`/`email_secondary` as plain optional string fields (not an item-based list) because the contact page markup has a fixed, small (2) count of each — an item architecture would be over-engineering for a fixed pair.
- **Scoping decision — "major content area" interpretation**: pages with multiple distinct prose blocks (e.g. About's Company Intro / Vision-Mission / EV Section) are still represented by a single `content` (richtext) section, not multiple `content` sections — `page_sections` enforces one row per `section_key` per page (`firstOrCreate(['section_key' => ...], ...)`), so multiple distinct prose blocks on one page are out of scope for "one section per content type" without inventing new section_keys. Only the 5 explicitly-specified new types (`hero-slide`, `stat`, `faq-item`, `process-step`, `checklist-item`) were added.

## Phase I notes (2026-06-15) — Frontend CMS Integration

- **Page resolution is centralized in one controller**: `App\Http\Controllers\PageController::show(string $slug, string $view)` is the single place that loads a `Page` by slug (404 if missing or not `published`), eager-loads `sections.items.fields.media` and `sections.fields.media`, and hands the view a `page` and a `sections` collection keyed by `section_key`. All 5 public routes are now one-line calls into this controller. Any future public page (e.g. a new landing page) should follow the same `app(PageController::class)->show($slug, $view)` pattern rather than adding a bespoke closure/controller.
- **`App\Cms\Content` is the only sanctioned way to read CMS content in public Blade views**: `field()`, `itemField()`, `items()`, `mediaUrl()`, `lines()`, `settingValue()`, `settingMediaUrl()`. Every converted page follows the same per-section pattern: `$xSection = $sections['section_key'] ?? null;` then `Content::field($xSection, 'key', '<original hardcoded value>')` for scalars and `Content::items($xSection, [...array of default item arrays matching the original hardcoded items...])` for repeatables. **Do not bypass this with direct `$section->field(...)` calls in new page conversions** — the helper centralizes the "missing/empty → fallback" logic that keeps the site from ever rendering blank content.
- **The `Content::items()` fallback rule is load-bearing and easy to break**: a section's real `section_items` are only used if `$items->isNotEmpty() && $items->contains(fn ($item) => $item->fields->isNotEmpty())`. This exists because `PagesSeeder` pre-creates *structural* `section_items` (zero `item_fields`) for every repeatable block on every system page, so a naive `isNotEmpty()` check would render real-but-empty items (blank carousel slides, blank cards) instead of the original copy. This rule operates per-section: as soon as *one* item in a section has *any* field, the whole section switches to "real items" mode, and any other items in that section that are still empty will render with `null`/empty field values (not the hardcoded array defaults) — there is no per-item fallback, only per-section. Keep this in mind if `PagesSeeder` or the admin UI ever start creating items with partial field sets.
- **Theme settings are recomputed per-request via a view composer, not shared once at boot**: `AppServiceProvider::boot()` registers `View::composer('*', fn ($view) => $view->with('themeSettings', $this->loadThemeSettings()))`, replacing the Phase D-era `View::share(...)` (computed once at app bootstrap). Reasons: (1) test correctness — `View::share` ran before `SettingsSeeder` in `setUp()`, so settings changes made mid-test were invisible; (2) production correctness on any persistent-worker setup (Octane, queue workers rendering views) — a one-time boot-time snapshot would never reflect admin edits to Theme Settings without a restart. The per-request query is `Setting::where('group','theme')->with('media')->get()->keyBy('key')` — cheap (14 rows) and wrapped in try/catch returning `collect()` if the table doesn't exist yet (fresh install before migrations).
- **Theme colors render as CSS custom-property overrides, not a recompiled stylesheet**: `resources/views/partials/theme-style.blade.php` emits `<style id="cms-theme-overrides">:root{--bs-primary: #...; ...}</style>` via `Content::settingValue($themeSettings, '..._color', '<default hex>')` for all 10 theme colors (primary/secondary/accent/success/warning/danger/dark/light + the existing button/link colors from Phase D). This works because the existing Bootstrap 5 + custom CSS already consumes `--bs-*` custom properties — no Vite rebuild or SCSS recompilation needed when an admin changes a color; it takes effect on the next page load. If the visual design ever moves away from CSS-custom-property-driven theming, this partial's approach would need rethinking.
- **`SettingsSeeder` extended for Phase I**: added `success_color` (#198754), `warning_color` (#ffc107), `danger_color` (#dc3545), `dark_color` (#212529), `light_color` (#f8f9fa) to the existing 9 Phase D colors, bringing the seeded `theme` group to 14 rows (12 colors + `logo`/`favicon` media references), matching the full Phase I color set requirement.
- **`Media::getUrlAttribute()` added**: `app/Models/Media.php` now has `public function getUrlAttribute(): string { return Storage::disk($this->disk)->url($this->path); }`, used by `Content::mediaUrl()`/`Content::settingMediaUrl()` and directly in tests (`$media->url`). Previously no accessor existed for resolving a `Media` row to a public URL.
- **Logo/favicon fall back to existing static assets**: `Content::settingMediaUrl($themeSettings, 'logo'|'favicon')` returns `null` if no `Media` is assigned (default fresh-install state), and `partials/head.blade.php`/`partials/navbar.blade.php`/`partials/footer.blade.php` fall back to the original hardcoded `asset('assets/img/...')` paths in that case. No "broken image" risk on a fresh install.
- **Richtext fields remain unsanitized**: `content.body` (and similar `richtext` fields) render via `{!! Content::field(...) !!}` (raw HTML), unchanged from before Phase I. This is a pre-existing gap (flagged since Phase F) — anyone with admin access to edit a page's `content` section can inject arbitrary HTML/JS into the public site. If the admin panel is ever opened to less-trusted editors, sanitize on save in `PageSectionController`/`SectionItemController` (e.g. `mews/purifier`), not at render time.
- **Known hardcoded blocks intentionally left as-is (Class B/C, unchanged from Phase E.1)**:
  - Home's second "Next-Generation Technology & EV Solutions" block (progress bars + video modal trigger) — no schema, fully hardcoded.
  - Home's "about" image collage (3 images) and "25/Years Experience" badge overlay — the `about` section's `image` field (single) and `stat` items (3 counters) are CMS-driven, but the collage layout/badge text itself is hardcoded markup.
  - About page's small "EV Section" icon/title cards — hardcoded.
  - `brand_logos` items have a `url` field in the schema (link target per logo) that is not rendered/used on the frontend — logos render as static images only, not links.
- **Test pattern for new public-page tests**: extend `Tests\TestCase`, `use RefreshDatabase`, and in `setUp()` run `(new SettingsSeeder())->run(); (new PagesSeeder())->run();` (see `tests/Feature/PublicPagesTest.php`). To test a CMS override, fetch the relevant `PageSection`/`SectionItem` via `Page::where('slug', ...)->firstOrFail()->sections()->where('section_key', ...)->firstOrFail()` and create a `section_fields`/`item_fields` row directly — no need to go through the admin HTTP endpoints. Watch for double-escaping when asserting on text containing `&`/HTML entities: `assertSee()` escapes its argument by default, so pass already-unescaped text (e.g. `&` not `&amp;`) unless also passing `false` as the second argument.

## Production Hardening Sprint 1 notes (2026-06-15)

- **SEO metadata architecture**: `resources/views/partials/head.blade.php` resolves 7 of the 8 `pages.*` SEO columns (`meta_title`, `meta_description`, `meta_keywords`, `canonical_url`, `robots`, `og_title`, `og_description`, plus `og_image_id` via the `ogImage` relation) through the new `App\Cms\Content::pageMeta(?Page $page, string $column, ?string $default = null)` helper — same `null`/`''` → `$default` pattern as `Content::field()`/`itemField()`. The `$page` variable is available inside `partials.head` because `@include` shares the parent view's data, and `PageController::show()` always passes `page` to the rendered view.
  - **Fallback chain**: per-page `@section('title'|'meta_keywords'|'meta_description', ...)` values (already present in every page view from before Phase I) are read via `$__env->yieldContent('title'|'meta_keywords'|'meta_description', '<built-in default>')` and used as the *fallback* when the corresponding `pages.meta_title`/`meta_keywords`/`meta_description` column is empty. This means: if an admin never touches a page's SEO fields, the site renders exactly the same `<title>`/meta tags as before this sprint; as soon as an admin fills in `meta_title` etc. via `admin/pages/edit.blade.php`, those values take over.
  - `robots` defaults to `'index, follow'` if empty (the `pages` table column itself already defaults to `'index,follow'` at the DB level, so this Blade-level fallback is a belt-and-suspenders case for rows created outside the normal flow).
  - `canonical_url` defaults to `url()->current()` (the current request's own URL) — every page is self-canonical unless an admin sets a different canonical.
  - `og_title`/`og_description` default to the resolved `$seoTitle`/`$seoDescription` (so Open Graph tags are never empty once title/description resolve to *something*).
  - `og:image` is only emitted if `$page->ogImage` is loaded and non-null — no broken/empty `og:image` tag on a fresh install. `PageController::show()` eager-loads `ogImage` (a `belongsTo(Media::class, 'og_image_id')`) alongside the existing `sections`/`items`/`fields`/`media` eager-loads, so this adds **zero** extra queries per request.
  - **If a future page type needs different SEO defaults** (e.g. a blog post template wanting `og:type = article`), branch on `$page->template` inside `partials/head.blade.php` rather than hardcoding `og:type = website` — currently all 5 system pages use `website` and there was no requirement to differentiate.

- **Theme-settings loading strategy — container-singleton-per-request, not `View::share`, not a persistent cache**: `AppServiceProvider::boot()` now does:
  ```php
  $this->app->singleton('cms.theme-settings', fn () => $this->loadThemeSettings());
  View::composer('*', fn ($view) => $view->with('themeSettings', $this->app->make('cms.theme-settings')));
  $this->app->terminating(fn () => $this->app->forgetInstance('cms.theme-settings'));
  ```
  - First view-composer invocation in a request resolves and caches the `Setting::where('group','theme')->with('media')->get()->keyBy('key')` query in the container; every subsequent `View::composer('*', ...)` invocation in the *same* request (head, topbar, navbar, theme-style, page content, footer, scripts, layout — ~8 view instances) reuses the cached `Collection` with **zero** additional queries (verified: settings query count per request is 1, down from 8).
  - The `terminating` callback runs after the response is sent (Laravel calls `$kernel->terminate()` at the end of every real request, and `Illuminate\Foundation\Testing\Concerns\MakesHttpRequests` calls it too, so this also works correctly across multiple `$this->get()` calls within a single PHPUnit test) — `forgetInstance('cms.theme-settings')` clears the cached value, so the *next* request/test re-runs the query and sees any settings changes made in between. This is why `test_theme_color_settings_render_as_css_overrides` (2 requests with a `Setting::update()` in between) still passes.
  - **Why not a persistent cache (`Cache::remember`)?** A TTL-based or manually-invalidated cache would reintroduce the "stale theme settings after an admin edit" problem the Phase I `View::share` → `View::composer` change was specifically designed to avoid (see Phase I notes above) — every admin Theme Settings save would need to know to bust the cache. The container-singleton-per-request approach gets the "only one query" win with **no invalidation logic at all**, because the cache's lifetime is exactly one request.
  - **Octane/persistent-worker caveat (documented, not yet relevant)**: this app does not currently use Laravel Octane (`composer.json` has no `laravel/octane`). If Octane is adopted later, `terminating` callbacks still fire per-request under Octane (Octane proxies the standard request lifecycle), so this pattern should continue to work without modification — but verify with Octane's `OCTANE_FLUSH`/container-reset behavior at that time, since some long-lived singletons need to be added to Octane's flush list explicitly.

- **RichText sanitization strategy and tradeoffs**: added `ezyang/htmlpurifier` (`^4.18`) — the project's first new Composer dependency since Phase B. `App\Cms\Content::richtext(?string $html): string` wraps an `HTMLPurifier` instance (lazily constructed once per request via a private static property on `Content`) configured with:
  - **Allowed tags/attributes**: `p[class]`, `br`, `strong`, `em`, `b`, `i`, `u`, `h1`-`h6[class]`, `ul[class]`/`ol[class]`/`li[class]`, `a[href|title|target|rel]`, `blockquote`, `span[class]`. This covers everything the 4 existing `content.body` hardcoded defaults use (`<p class="mb-4">`, `<strong>`) plus headings/lists/links/blockquotes for future admin-entered content.
  - **`HTML.TargetBlank` enabled**: any `<a>` tag automatically gets `target="_blank" rel="noreferrer noopener"` — prevents `rel`-less `target="_blank"` reverse-tabnabbing if an admin pastes a link.
  - **Stripped by design**: `<script>`, inline event handlers (`onerror`, `onclick`, `onmouseover`, etc. — HTMLPurifier strips all `on*` attributes since they're not in `HTML.Allowed`), `javascript:`/`data:` URI schemes in `href`/`src` (HTMLPurifier's default `URI.AllowedSchemes` is `http`/`https`/`mailto`/`tel`/etc., not `javascript`), `<img>`/`<iframe>`/`<style>`/`<form>` and any other tag not explicitly allowlisted.
  - **Where it's applied**: render-time only, via `{!! Content::richtext($contentBody) !!}` in `resources/views/pages/{about,services,solutions,contact}.blade.php` (the only 4 places `content.body`/`richtext` fields are output with `{!! !!}`). **Not** applied at save-time in `PageSectionController`/`SectionItemController` — chosen deliberately so that (a) sanitization rules can be tightened later without a data-migration/re-save step, and (b) any already-stored unsafe HTML (from before this sprint) is also covered retroactively without a backfill.
    - **Tradeoff**: every page render re-purifies the same stored HTML. For 4 pages × 1 `content.body` field each, this is negligible (~1-2ms after HTMLPurifier's definition cache is warm). If `richtext` fields become numerous/large, consider switching to save-time sanitization (sanitize once in `HandlesTemplateFields::syncTemplateFields()` before `updateOrCreate`) plus a one-time backfill command — but render-time was the right call for the current scale and for retroactively covering existing content.
  - **Definition cache**: `Cache.SerializerPath` points at `storage/framework/cache/purifier/` (new directory, `.gitignore`'d like `storage/framework/cache/data`). HTMLPurifier writes its compiled HTML/CSS/URI definition there on first use; without a writable cache dir it still works but recompiles the definition on every request.
  - **Not addressed (future work)**: `richtext` fields in `config/cms/templates.php` are not validated against this allowlist at *admin save time* — an admin can still type/paste arbitrary HTML into a `<textarea>` and it will be stored verbatim; it's only sanitized when rendered on the public site. This is intentional (see "where it's applied" above) but means the admin "preview" (if one is ever added) should also run content through `Content::richtext()` to match what the public site will actually show.

## Phase G notes (2026-06-16) — Menu Management CRUD

- **Two singletons now in AppServiceProvider**: `cms.theme-settings` (pre-existing) and `cms.menus` (new). Both follow the identical pattern: singleton registered in `boot()`, resolved by `View::composer('*',...)`, cleared by `terminating()`. Any future per-request shared data (e.g. a global announcements banner) should follow this same three-part pattern rather than using `View::share()` (boot-time snapshot, wrong for test isolation) or re-querying in every view composer invocation (N+1).
- **`Menu::items()` relationship scope and `create()` behavior**: The `items()` relationship is defined as `$this->hasMany(MenuItem::class)->whereNull('parent_id')->orderBy('order_column')`. When calling `$menu->items()->create([...])`, Laravel only sets `menu_id` automatically — it does NOT set `parent_id=null`. Since `parent_id` is a nullable DB column, it defaults to null. `MenuItemController::store()` passes `'parent_id' => null` explicitly in the create array to make intent clear and ensure Phase G items are always top-level.
- **`Menu::items()` scope and the `move()` algorithm**: The `$menuItem->menu->items` load returns the Menu's `items` collection scoped to `whereNull('parent_id')->orderBy('order_column')`. This is the correct sibling set for the adjacent-swap algorithm — only top-level items, already sorted by display order. Children (out of Phase G scope) are excluded and unaffected by moves.
- **Soft-deleted page references in menu items**: A menu item can reference a page that is later soft-deleted. The FK cascade (`cascadeOnDelete`) only fires on hard-delete of the page. After soft-delete, `$menuItem->page` returns null. All frontend menu rendering uses `$item->url ?? ($item->page ? route($item->page->slug) : '#')` to guard against null. The validation rule `Rule::exists('pages','id')->where(whereNull('deleted_at'))` prevents creating NEW items against soft-deleted pages, but does not retroactively remove existing items when a page is soft-deleted.
- **Footer "Popular Links" column removed (Option A, locked decision)**: The footer previously had two identical dead-link columns ("Quick Links" and "Popular Links"). Only one footer menu exists (`key='footer'`). "Popular Links" was removed and will not return. If a second distinct footer column is ever needed, it requires a new menu key + seeder entry + the column restored — not a future migration, just a config/seeder/Blade change.
- **is_active filter is NOT on the relationship, only on frontend load**: `Menu::items()` has no `is_active` filter — it returns all top-level items (active or inactive). This is intentional: the admin edit view needs to show inactive items so the admin can re-enable them. The frontend loading query (inside `loadMenus()` in AppServiceProvider) adds `->where('is_active', true)` in the eager load closure, so only active items are passed to `$headerMenu->items` and `$footerMenu->items` in Blade.

If an "archive without reading" bulk action is ever added (e.g. archiving a batch of obvious spam without opening each one), revisit this — at that point "archived + unread" becomes a legitimate state and the dashboard's unread count / `unread()` scope would need to decide whether archived-unread messages count as "unread".
- **`contact.submit` and `appointment.submit` now have `throttle:10,1`** (10 requests/minute, keyed by IP — Laravel's default for unauthenticated routes). Chosen as more permissive than `admin.login.submit`'s `throttle:5,1` since these are legitimate public-form routes (a real visitor could plausibly retry after a validation error a few times), not auth-attempt routes where aggressive throttling deters brute-forcing. 10/min still bounds automated abuse/spam without a CAPTCHA. If spam becomes a real problem, layer a honeypot field or CAPTCHA on top rather than tightening this further (a legitimate user hitting a 429 on a contact form is a bad UX outcome).
- **Empty-string `phone`/`subject` are normalized to `null` in `ContactMessageController::store()`** before `ContactMessage::create()`. Both fields are `nullable` in validation, but an HTML form always POSTs every field (even empty text inputs), so without this normalization `''` (not `null`) would be stored whenever a user leaves these blank. The admin views use `$contactMessage->phone ?? '—'` / `?? '—'`-style placeholders, which only work correctly for `null` — an empty string would render as a blank cell instead of the placeholder. This normalization makes "field omitted entirely" and "field submitted blank" behave identically.
- **Mass-assignment of `source_page`/`ip_address`/`status` is structurally impossible**, not just validated-against: `ContactMessageController::store()` spreads `$validated` (which only ever contains `name`/`email`/`phone`/`subject`/`message` — the 5 validated keys) and then unconditionally overwrites `source_page`/`ip_address`/`status` as the last three array keys passed to `create()`. Even if `$validated` somehow contained those keys, the later keys win. `test_contact_form_submission_cannot_override_source_page_status_or_ip` locks this in.
- **Dashboard count tests now assert on view data (`assertViewHas('totalMessages'|'unreadMessages', N)`), not rendered text** — the old `assertSee('2')` was a false-positive risk because the dashboard's (still-hardcoded) "Total Pages" card also renders `12`/other digits that could coincidentally contain `'2'`. `assertViewHas` reads the actual values `DashboardController::index()` passed to the view, so the test fails if the underlying `count()`/`unread()->count()` queries are wrong, independent of Blade rendering.
- **New stored-XSS regression test for admin contact-message views** (`test_admin_views_escape_message_content_against_stored_xss`) mirrors the existing `PublicPagesTest::test_richtext_content_is_sanitized_against_stored_xss` pattern but for `admin/contact-messages/{index,show}.blade.php`. These views use Blade's default `{{ }}` escaping (not `{!! !!}`), so a `<script>` in a submitted `name`/`message` renders as `&lt;script&gt;...` — this test guards against a future edit accidentally switching to `{!! !!}` for these fields.
- **Deferred (Nice-to-Have, not in Phase H.1 scope)**: duplicated status-badge `match` expression across `admin/contact-messages/index.blade.php` and `show.blade.php` (extract to a Blade component or `ContactMessage` accessor if a third place needs it); default Tailwind pagination view rendering inside the Bootstrap-styled admin (`admin/contact-messages/index.blade.php` and `admin/media/index.blade.php` both affected — would need `php artisan vendor:publish --tag=laravel-pagination` + a Bootstrap pagination view); `status` remains a plain `string` column rather than a PHP enum/DB enum — fine at 3 known values (`new`/`read`/`archived`) but worth revisiting if more statuses are added; `(status, created_at)` composite index not added — current message volume doesn't warrant it.

## Tiptap admin editor — selector and timing fixes (2026-06-18)

- **Never use `document.querySelector` + `CSS.escape()` to find a field's textarea/toolbar by `name`/`data-toolbar-for` attribute.** Field names like `fields[body]` need bracket-escaping, and the escaped CSS attribute selector silently matched the wrong/no element in practice (cause not fully root-caused, but reproducible: the Tiptap editor never mounted, leaving the static server-rendered HTML visible — which *looks* like "read-only text with no toolbar" but is actually "Tiptap never initialized at all"). Fix: scope lookups via `el.closest('.richtext-wrapper')?.querySelector('textarea')` / `?.querySelector('.richtext-toolbar')` — the textarea/toolbar are always siblings of the `[data-richtext]` div inside the same `.richtext-wrapper`, so DOM traversal needs zero name-matching and can't pick the wrong element even when two sections share a field key (e.g. `hero.body` and `ev_solutions.body` both render `name="fields[body]"`).
- **`show.bs.collapse` fires *before* Bootstrap removes `display:none`** (it only flips the element to `display:block; height:0` partway through `_enter()`); ProseMirror initializing at that point can render incorrectly. **`shown.bs.collapse`** fires after the expand animation completes, with real layout — use that for any JS widget (Tiptap, charts, anything needing `getBoundingClientRect()`) that lazily initializes inside a Bootstrap collapse.
- **Nested collapses (item-body inside section-body) need an explicit skip-guard**, not just lazy init: when a section's `shown.bs.collapse` fires, `e.target.querySelectorAll('[data-richtext]')` also matches editors inside still-closed item panels nested within it. Mounting those early means the per-element "already mounted" `WeakSet` guard prevents them from ever getting a clean re-mount when their own item panel opens later. Guard with `!el.closest('.collapse:not(.show)')` at the point of mounting, both in the `shown.bs.collapse` listener and in any manual "restore accordion state from sessionStorage" code path — only mount what's now actually visible, let each nested panel's own `shown.bs.collapse` handle the rest.

## Section-type reuse across templates — `hero` vs `page_header` (2026-06-18)

- **A section type's schema (especially its `items`) is shared by every template that lists it in `allowed_sections` — the admin UI has no per-template override.** `hero` was reused by all 5 page templates: home actually renders its 3 `hero-slide` items as a real carousel, but `about`/`services`/`solutions`/`contact` only ever read `hero`'s top-level `heading`/`subheading`/`background_image` fields for a static banner and never touch `hero`'s items. Because `admin/pages/edit.blade.php` renders "Add Item" controls based on `TemplateRegistry::section($key)['items']` (the type's schema), every non-home page inherited a fully-functional "Add Slide" button that does nothing on the public site if used — confusing, not a hard bug, but a real admin/frontend structural mismatch.
- **Fix pattern for this class of problem**: don't try to suppress items conditionally per-template (the registry has no such concept and adding one would be a bigger structural change for a one-off case) — instead, split into two section types with the same view name and overlapping fields (`hero` keeps `items`/`autoplay`/`interval` for the real carousel; new `page_header` is `heading`/`subheading`/`background_image` only, `items => null`) and point each template's `allowed_sections` at whichever one matches what it actually renders. Existing DB rows need a data migration (`UPDATE page_sections SET section_key = 'page_header' WHERE section_key = 'hero' AND page_id IN (non-home pages)`), not just a code/seeder change — otherwise already-edited admin content for those pages becomes orphaned (the page would silently fall back to hardcoded Blade defaults since `$sections['page_header']` would be null while the real data sits under the old `hero` key).
- **Tell**: if you're ever asked "why does the admin show controls for X that don't do anything on the frontend," check whether the section/item *type* is shared across templates and whether all consumers of that type actually use every part of its schema — `TemplateRegistry::section($key)['items']` is type-level, not page-level.

## `url` field type silently blocked re-saving relative paths / mailto: links (found + fixed 2026-06-18)

- **The bug**: `HandlesTemplateFields::templateFieldRules()` mapped `type => 'url'` straight to Laravel's built-in `'url'` validation rule, which requires a fully-qualified URL with scheme (`https://...`). But `ContentSeeder` — and any admin typing a normal internal link — routinely stores **site-relative paths** (`/contact`, `/about`, `/solution`) and `mailto:`/`tel:` links in `button_url`/`link_url`/`map_embed_url` fields, because that's the natural way to link to another page on the same site or an email/phone action. Seeding bypasses validation (`updateOrCreate` direct on the model), so this was invisible until an admin tried to **re-save** a section that already had one of these values — at which point the whole form's save would fail with a validation error on a field the admin never even touched.
- **Confirmed scope**: 6 real seeded values were affected sitewide (`/contact` ×3 in various `cta.button_url`, `mailto:info@poisedtechnology.com` in contact's `cta.button_url`, `/solution` and `/about` in home's hero slide `button_url`s) — i.e. almost every page's CTA section was one accidental re-save away from becoming stuck.
- **The fix**: replaced the `'url'` rule with a closure (`HandlesTemplateFields::urlOrPathRule()`) that accepts an absolute URL (`filter_var(..., FILTER_VALIDATE_URL)`), OR a string matching `^(/[^\s]*|mailto:.+|tel:.+)$`. Wrapped in `Rule::when(filled(...), [...], ['max:255'])` so empty/null values for non-required fields still pass cleanly.
- **Takeaway**: any TemplateRegistry field typed `'url'` is meant for "things you can link to" — which on this site routinely includes internal relative paths and `mailto:`/`tel:`, not just external absolute URLs. Don't revert to a strict `'url'` Laravel rule for this field type. If a *truly* external-only URL field is ever needed (e.g. a "Verified by" badge linking off-site), that would need its own stricter field type rather than changing this shared one back.
- See [[notes.md]] "PagesSeeder order_column was not self-healing" above for the related discovery that seeder-written content bypasses admin-form validation entirely — that's exactly why this bug stayed invisible for as long as it did.

## Shared section types: a field only counts as "editable" if EVERY consuming page's view renders it (2026-06-19)

- **The general bug class**: a section type's *schema* (`config/cms/templates.php`) is shared by every page template that lists it in `allowed_sections`. The admin (`admin/pages/edit.blade.php`) renders an input for every field in that schema regardless of which page is open. But each public page's Blade view independently decides which of those fields to actually call `Content::field()`/`Content::itemField()` for. **If a field is only read by one of N consuming pages' views, then editing it on any of the other N-1 pages is a silent no-op** — indistinguishable, from the admin's perspective, between "saved successfully but does nothing" and "didn't save." This was first hit with `hero`/`page_header` (fixed 2026-06-18), then found to be systemic across 6 more section types on 2026-06-19 (`ev_solutions`, `content`, `services_grid`, `stats`, `checklist`, `cards`) — see the matching project_progress.md Change Log entry for the full list.
- **Decision framework for fixing an instance of this** (used across both fix passes):
  1. **Render-everywhere** — if the field is plain text/data with no layout dependency (a heading, a link, a description line), just add the missing `Content::field()`/`Content::itemField()` call to the page(s) missing it, wrapped in `@if` so the default appearance is unchanged unless an admin actually sets it. Cheapest fix, no migration, no schema change. Used for `stats.heading`/`subheading`, `checklist.description`, `cards.heading`/`subheading`, `services_grid.highlights`/`link_text`/`link_url`.
  2. **Section-type split** — if the field needs real layout space the other page's design doesn't have (an overlay badge needs a position-relative wrapper; a background video needs a `<video>` element that doesn't exist on that page at all), split the section type: keep the full schema on the page(s) that need it, add a new leaner type (reusing the same item schema where applicable) for the page(s) that don't. Requires: a data migration renaming the affected page's existing `page_sections.section_key` (preserve existing field/item data — never just re-seed and orphan it), a `ContentSeeder`/`PagesSeeder` key update, and an `allowed_sections` swap. Used for `ev_solutions`→`ev_highlights` (About) and `content`→`about_intro` (About).
  3. **Remove the field entirely** — if NO consuming page has anywhere to render it (not "one page does, others don't" but "none of them do"), it shouldn't be a field at all; remove it from the schema and clean up any orphaned DB values via the same migration. Used for `checklist.heading`.
- **Before adding any new field to a section type already used by ≥2 templates**, check every consuming page's Blade view actually renders that field — don't assume "it's optional, so it's harmless if unused." An unused-but-harmless field (e.g. `brand_logos.brand-logo.url`, intentionally not rendered anywhere, documented as a known gap in Phase I notes) is a one-time, explicit, documented decision — it is NOT the same as accidentally reintroducing this bug class via reflexive reuse. When in doubt, spawn a research pass across all consuming Blade views before deciding reuse is safe (see [[project_progress.md]] 2026-06-19 entry for an example of that audit).
- **Tell**: if a user reports "the admin page doesn't match the frontend" or "I filled in X but it doesn't show," check whether the section being edited is a shared type and whether the specific field in question is actually read by that exact page's Blade view — don't assume the bug is in `Content::field()`/rendering order/caching first.

## Local environment — actual DB driver mismatch vs CLAUDE.md (2026-06-18)

- **This environment's `.env` has `DB_CONNECTION=mysql` (host `127.0.0.1:3306`, database `poised_cms`) via a local XAMPP MySQL install — NOT sqlite**, despite `CLAUDE.md`/earlier notes documenting SQLite (`database/database.sqlite`) as the default. The sqlite file still exists on disk (harmless leftover) but is not what `php artisan` commands or the dev server actually read locally. **Always check `.env`'s `DB_CONNECTION` before assuming which database a local command is hitting** — don't trust the documented default over the actual file.
- **`php artisan migrate`/`db:seed` against this MySQL connection can hang indefinitely with ~0% CPU if `mysqld.exe` isn't actually listening** (e.g. mid-restart, or the user believes XAMPP is uninstalled but the process is still alive/dead in an inconsistent state) — there's no connection timeout configured, so a dead/unreachable MySQL server manifests as a silently stuck artisan command, not a fast connection-refused error. If an artisan DB command seems to hang with no output, check `Get-CimInstance Win32_Process -Filter "Name='mysqld.exe'"` (or equivalent) before assuming a SQLite lock or other cause.
- **All current content in this DB is 100% seeder-reproducible** (`SettingsSeeder` → `PagesSeeder` → `MenusSeeder` → `ContentSeeder`, called by `DatabaseSeeder`) — confirmed by fully recovering from a real data-loss incident (XAMPP MySQL `poised_cms` schema wiped to zero tables) via `php artisan migrate --force && php artisan db:seed --force` alone, no manual re-entry. **This stops being true the moment an admin makes a manual edit through the Pages/Media/Menu/Settings admin UI** that isn't mirrored in a seeder — at that point, losing the DB again means losing real, non-reproducible work, and a real backup strategy (`mysqldump`, scheduled or pre-deploy) becomes necessary.
- **`php artisan db:seed --force` (the full `DatabaseSeeder`) is NOT safe to re-run against a database that already has the seeded user** — `User::factory()->create([...])` in `DatabaseSeeder::run()` is unconditional (not `firstOrCreate`), so a second full run throws a unique-constraint violation on `users.email`. To re-apply only structural/content seeders (e.g. after a config change adds a new section), run the specific seeder classes directly: `php artisan db:seed --class=PagesSeeder --force` then `--class=ContentSeeder --force` — both of those (and `SettingsSeeder`/`MenusSeeder`) are fully idempotent via `firstOrCreate`/`updateOrCreate`, unlike `DatabaseSeeder` itself.

## `PagesSeeder` `order_column` was not self-healing on reorder (found + fixed 2026-06-18)

- **The bug**: `PagesSeeder`'s per-page loop did `$page->sections()->firstOrCreate(['section_key' => $sectionKey], ['order_column' => $index, ...])` — `order_column` was only ever set at *creation* time. If a template's `allowed_sections` array in `config/cms/templates.php` is later reordered or has a new section inserted in the middle (exactly what happened when `ev_solutions` was added to `standard_page.allowed_sections` between `cards` and `stats`), the **already-existing** sections keep their stale `order_column` values from whenever they were first created, while the **new** section gets the index it has in the *current* array — which can tie with or land after sections it should visually precede. Confirmed concretely: inserting `ev_solutions` at array-index 4 collided with the already-existing `stats` section (also `order_column=4`, assigned back when `stats` was at index 4 in an older, shorter `allowed_sections` list), and `Page::sections()`'s `orderBy('order_column')` broke the tie in a MySQL-internal order that put `ev_solutions` *after* `stats` instead of before it.
- **The fix**: `PagesSeeder` now re-syncs `order_column` to the section's current array index on *every* run, not just at creation (`if ($section->order_column !== $index) { $section->update(['order_column' => $index]); }`), right after the `firstOrCreate` call. This makes section ordering self-healing the next time `PagesSeeder` runs after any future `allowed_sections` reorder/insert — no more manual order_column fixups needed.
- **Takeaway for future schema changes**: any time a new section type is inserted into the *middle* of an existing template's `allowed_sections` (as opposed to appended at the end), re-run `php artisan db:seed --class=PagesSeeder --force` afterward and spot-check the affected page's section order (`$page->sections->pluck('section_key')`) — don't assume the array order in config automatically reflects in the DB for pages that were seeded before the config change.

## `@section('name', null)` silently destabilizes Blade's output-buffer bookkeeping (found + fixed 2026-06-20)

- **The bug**: `pages/products/show.blade.php`, `pages/blog/show.blade.php`, and `pages/news/show.blade.php` all used the pattern `@section('meta_description', $model->meta_description ?? $model->excerpt)` (or `short_description`). When **both** operands are `null` — a realistic state for a freshly created Product/BlogPost/NewsArticle before an admin fills in either field, not just a contrived test case — this compiles to `$__env->startSection('meta_description', null)`. PHPUnit's risky-test detector caught it as "Test code or tested code did not close its own output buffers" on the affected response assertions; the response itself still rendered with HTTP 200 and all visible content correct, so this is a **silent** failure mode that automated content assertions alone would never catch.
- **How it was root-caused**: bisection, not guessing — stripped the affected Blade view down to a minimal reproduction (`@extends` + a single `{{ $post->title }}`), confirmed the risky flag disappeared, then added blocks back one at a time (page-header, richtext body, tags/share buttons, prev/next, related-posts) until the flag reappeared. It reappeared the moment `@section('meta_description', ...)` was added back with both fallback operands null — every other block in the view (including `Content::richtext()`/HTMLPurifier, initially suspected) was innocent.
- **The fix**: always add a final non-null fallback — `$model->meta_description ?? $model->excerpt ?? ''` — so `@section()` is never called with a literal `null`. Applied to all three `show.blade.php` files.
- **Takeaway**: any `@section('name', $maybeNull)` call (as opposed to the block form `@section('name') ... @endsection`) must guarantee its value is never `null` — an empty string is safe, `null` is not. This is easy to miss because the symptom is a PHPUnit "risky" flag (not a hard failure) rather than a visible bug, and only manifests when *every* fallback in the `??` chain is empty — exactly the kind of edge case that "looks fine in manual testing with realistic seeded data" but breaks for a genuinely empty new record. Grep for `@section('` with a literal second argument (not a block) before assuming a `??` chain is safe — check what happens if every operand is null/empty, not just the happy path.

## `ContentRevisionService` must not assume every revisionable model has `HasContentMedia` (found + fixed 2026-06-20)

- **The bug**: `ContentRevisionService::record()`/`snapshot()`/`restore()` (the generalized Product/BlogPost/NewsArticle equivalent of `PageRevisionService`) was written against Product/NewsArticle, both of which `use HasContentMedia` for gallery/document attachments, and unconditionally called `->with(['gallery.media', 'documents.media'])` and `$model->gallery`/`$model->documents`. `BlogPost` deliberately does **not** use `HasContentMedia` (the spec only asked for inline body images via Tiptap, not a separate gallery/downloads section) — saving any blog post threw `Call to undefined relationship [gallery] on model [App\Models\BlogPost]`. Caught immediately by a freshly-written feature test (`BlogPostManagementTest::test_authenticated_user_can_create_a_post`), not by manual exploration — a concrete argument for writing the CRUD test for *every* new resource immediately, even ones that feel like simple copies of an already-working one.
- **The fix**: `ContentRevisionService::usesContentMedia(Model $model): bool` checks `in_array(HasContentMedia::class, class_uses_recursive($model), true)` and gates all three gallery/document-touching code paths (eager-load in `record()`, the `gallery`/`documents` keys in `snapshot()`, the restore loop in `restore()`) behind it.
- **Takeaway for any future resource added to this revisioning system**: `ContentRevisionService` is meant to work for *any* `Model`, with or without `HasContentMedia` — don't add a new unconditional `$model->someRelation` call to it without the same `usesContentMedia()`-style guard (or an equivalent capability check), even if every resource that exists *today* happens to have that relation.

## Settings page had the exact same "schema vs. consumer" bug class as CMS sections — plus a genuinely-missing scope (found + fixed 2026-06-20)

- **The bug, restated for Settings**: `/admin/settings` only exposed 9 of `SettingsSeeder`'s 14 seeded theme colors (`success_color`/`warning_color`/`danger_color`/`dark_color`/`light_color` had no form field *or* validation rule in `SettingController::COLOR_KEYS` — permanently stuck at seeded defaults, no way to ever change them). The other 6 colors (`header_background`, `footer_background`, `button_color`, `button_hover_color`, `text_color`, `link_color`) were the *inverse* problem: editable and saved fine, but `theme-style.blade.php` never read them, so saving was a silent no-op. This is the identical bug class already documented above as "Shared section types: a field only counts as editable if every consuming page's view renders it" (2026-06-19) — just discovered again in a different subsystem (Settings, not CMS page sections). **Tell for next time**: any time a project has a "config schema → admin form → consumer renders it" pipeline (template field schemas, settings keys, anything similar), check both directions independently — does every *seeded/schema* key have a form field, and does every *form* field have a real consumer? Don't assume symmetry.
- **The missing-scope half**: the page was literally only "Theme Settings" (colors/logo/favicon) despite the admin sidebar/page title implying full site configuration, while equivalent contact/social/SEO/footer content was hardcoded redundantly in 3 different public partials (`topbar`, `navbar`, `footer`) that could silently drift from each other. **Tell**: a "Settings" page that covers less than what's hardcoded directly in the public templates it's supposed to control is a sign the settings table/page grew piecemeal (one feature at a time) without ever being audited against what's actually hardcoded sitewide. Worth periodically grepping the public partials for hardcoded contact info/social links/copyright text and checking whether each one has (or should have) a settings-backed equivalent.
- **Widening a globally-shared view variable is safe when the consumer helper is a pure keyed lookup with defaults**: `AppServiceProvider::loadThemeSettings()` went from `Setting::where('group','theme')` to `Setting::all()` (still shared as `$themeSettings` to every view via `View::composer('*', ...)`) with zero changes needed anywhere that already consumed it, because `Content::settingValue()`/`settingMediaUrl()` only ever do `$settings->get($key)` with a `$default` — extra keys present in the collection are simply never looked at by code that doesn't ask for them. This is a reusable pattern: if a shared collection's consumers are all pure-lookup-with-default, widening what's *in* the collection is safe; it would NOT be safe if any consumer iterated the whole collection assuming it only contained one kind of thing.
- **Curl-based admin login for manual verification is fragile across multiple sequential commands**: scraping `_token` from one `curl -c cookies.txt` response and feeding it to a *later*, separate `curl -b cookies.txt -c cookies.txt` call breaks if anything in between (or a stale `cookies.txt` from an earlier unrelated curl session in the same `/tmp`) caused the session/token pairing to drift — manifests as a `419 Page Expired` even though the form/route/CSRF middleware are all correct. Don't burn time debugging the app over this: for manual verification of admin-only behavior, it's faster and just as rigorous to (a) trust the automated feature-test suite's `actingAs()` coverage for the controller/validation/save logic, and (b) verify the *public-facing effect* of an admin change directly via `php artisan tinker` (update the DB row the same way the controller would) + a no-auth `curl` GET — exactly the pattern used to verify this Settings change.

## Soft-deleted `pages` rows still occupy the `slug` unique index (found + worked around 2026-06-21)

- **The bug**: `Page` uses `SoftDeletes`, but the `pages.slug` unique index is a plain MySQL unique constraint — it has no idea about `deleted_at` and does not exclude soft-deleted rows. Manual test/verification pages created and then soft-deleted earlier in a session (e.g. `Page::create(['slug' => 'career', ...])` followed by `$page->delete()` during an ad-hoc manual check) leave the slug permanently unavailable for `firstOrCreate()`/`create()` — Eloquent's *read* side correctly hides soft-deleted rows (`Page::where('slug', 'career')->first()` returns null, making it look safe to reuse the slug), but the *write* side still hits a real `UniqueConstraintViolationException` on insert, since the soft-deleted row's `slug` value is still physically in the table.
- **Where this bit us**: `StandalonePagesSeeder` tried to create a page with `slug => 'career'`; three slugs (`career`, `terms`, `faq`) silently collided with soft-deleted leftovers from an earlier "manually create a test page, verify, then delete it" verification pass in this same project, days/sessions apart.
- **The fix**: `Page::withTrashed()->whereIn('slug', [...])->forceDelete()` to physically clear the conflicting rows before re-seeding. There was no FK-cascade issue in this case (the leftover pages had no sections/items ever populated), but if a soft-deleted page DOES have real children, force-deleting it could orphan rows depending on whether those child tables' foreign keys cascade — check before force-deleting a page that might have real section/item history.
- **Takeaway**: after any manual "create a throwaway page via tinker/seeder to verify behavior, then delete it" step, prefer `forceDelete()` over the default soft `delete()` for genuinely-throwaway verification data — a soft-deleted row is invisible to normal queries but still a live landmine for any future `create()`/`firstOrCreate()` reusing the same slug. If a soft delete is unavoidable (e.g. testing the delete UI flow itself), remember to force-delete the residue afterward as part of cleanup, not just confirm it's hidden from listings.

## Git Bash on Windows silently mangles `/`-leading curl arguments into Windows paths (found 2026-06-21)

- **The trap**: this environment's Bash tool runs Git Bash (MSYS2), which auto-converts any command-line argument that *looks* like a Unix absolute path (starts with `/`) into a Windows path before the target program ever sees it — including values passed to `curl -d`/`--data-urlencode`. Posting `--data-urlencode "url=/products/verify-charger"` actually sent `url=C:/Program Files/Git/products/verify-charger` to the server. The symptom looked exactly like an application bug: a brand-new validation rule (`urlOrPathRule()`, written to *accept* relative paths) appeared to still reject one, and a direct `Validator::make()` reproduction in tinker with the literal same rules and the literal same string passed cleanly — strongly suggesting the rule was fine and something about the real request differed.
- **How it was actually root-caused**: temporarily logged the raw `$request->all()` inside the controller, made one real request, and read the logged value back — it was the mangled Windows path, not what was typed. Removed the debug log immediately after confirming.
- **The fix**: prefix the curl invocation with `MSYS_NO_PATHCONV=1` (e.g. `MSYS_NO_PATHCONV=1 curl ...`) whenever a request body/query/header value needs to start with a literal `/` and survive intact.
- **Takeaway**: if a manual curl-based verification produces a validation error that contradicts a clean isolated `Validator::make()`/tinker reproduction of the exact same rule and input string, suspect the shell mangled the argument before curl ever sent it — don't assume the validation rule itself is wrong just because a real HTTP request fails differently than calling the rule directly. This is a Windows/Git-Bash-specific gotcha; it would not occur on Linux/macOS or inside PowerShell.

## Any page with more than one `<form>` sharing field names needs named error bags, not the default one (found + fixed 2026-06-21)

- **The bug class**: Laravel's default validation error bag and `old()` input are both scoped to the **session**, not to an individual form or route. `/admin/menus/{id}` renders one `<form name="label"/"url"/"page_id"/"target">` per existing menu item plus one more for "Add Menu Item" — six forms, all reusing the same field names. A validation failure on *any one* of them flashes errors/old-input under the *same* keys, so every other form's `@error('label')`/`old('label', ...)` lit up too: a failed "Add Item" submission made every already-saved item appear to have validation errors, and silently blanked their Label inputs (since `old('label')` returns the flashed empty string the moment the key exists at all — it can't tell which form posted it).
- **The general fix**: give every form on a multi-form page its own named error bag (`$request->validateWithBag('uniqueName', [...])` server-side, `@error('field', 'uniqueName')` in Blade) — a static name for a form that only ever appears once per page (e.g. an "Add Item" form), a name derived from the record's ID for a form that repeats once per row (e.g. `'menuItem' . $item->id`). **`validateWithBag()` only scopes error *messages*** — it's confirmed (by reading `ValidatesRequests::validateWithBag()`) to be a thin wrapper that just sets `$e->errorBag` on the same `ValidationException`; the framework's `withInput()` still flashes the whole request as one flat global array regardless. So `old()` needs its own guard: only consult `old($key, $default)` when *that specific form's own bag* has errors (`$bag->any()`); otherwise use the real saved value directly. A one-line closure (`$oldFor = fn ($bag) => fn ($key, $default = null) => $bag->any() ? old($key, $default) : $default;`) made this a drop-in replacement everywhere `old(...)` was previously called bare.
- **Tell for next time**: any time a Blade view has more than one `<form>` element using the *same field names*, check whether each one's controller action validates into a named bag — if they all just call `$request->validate(...)` with no bag, a failure in one will visually contaminate every other form on the same page the next time it's rendered. This applies even if the forms post to different routes/controllers entirely (it's not about which controller handles it — it's about the *session* being shared across every form rendered in the same response). Every other admin edit/create screen in this codebase is single-form-per-page, so this class of bug couldn't occur there — it's specific to "list of independently-editable rows on one page," which currently only exists for Menu Items (and would recur for any future feature shaped like it: reorderable lists with an inline edit form per row, e.g. if Product features/specs ever grew an "edit existing row" form instead of pure client-side JS rows).

## `field-input.blade.php` already had a half-built, never-reachable `'icon'` field type (found 2026-06-21)

- **What was found**: while building the icon picker, `field-input.blade.php`'s `@switch($type)` already had a `@case('icon')` branch (live preview span + `data-icon-input`/`data-icon-preview` text input) and `admin-editor.js` already had a matching `initIconPreview()` listener for it — but every single icon field definition in `config/cms/templates.php` was still `'type' => 'string'`, so that branch could never execute. It was finished-looking dead code: no syntax error, no test failure, just silently unreachable.
- **Takeaway**: before building a new field-type case (or any "looks like it should already exist" feature) from scratch, grep the relevant `@switch`/`match` block for the type name first — there's real precedent in this codebase of a feature being partially wired on one side (the renderer) and never finished on the other (the schema that would actually select it). Finishing an existing half-built case is both less work and less risky than writing a parallel new one, but it's only possible if you look for it first.
